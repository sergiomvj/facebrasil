00 — Visão do Projeto

Por que este projeto existe

"O time de vendas não deve garimpar clientes — deve fechar negócios. O garimpo, a validação, o aquecimento e a qualificação são trabalho de máquina. SQLs chegam prontos, com contexto completo, no FBR-Click."

FBR-Leads é a plataforma que automatiza o ciclo completo de prospecção outbound: captação → enriquecimento → validação → aquecimento → handoff. Seis times de agentes OpenClaw trabalham em paralelo — cada agente em uma única tarefa por vez — para entregar somente SQLs ao time comercial: leads que já demonstraram interesse, com histórico de interações e dados completos.

O sistema segue integralmente a Bíblia FBR v1.0: agentes OpenClaw com 7 Markdowns versionados em Git, orquestração via n8n dedicado, cascata Ollama → Claude → GPT-4o, mail server Postal self-hosted e toda interação humana centralizada no FBR-Click. O vendedor nunca precisa abrir o dashboard — SQLs chegam como deals com @menção direta.

02 — Pipeline de Leads

Como um lead percorre o sistema

Todo lead segue um pipeline determinístico com descarte automático em qualquer etapa que falhe na validação. Apenas leads que passam por todas as etapas chegam ao time de vendas — nunca o contrário.

01
Captação
Scraping de sites institucionais, LinkedIn via Apify, CNPJ.biz, Google Maps e monitoramento de gatilhos de compra (vagas abertas, rodadas, expansões). Deduplicação automática por CNPJ/domínio.
Time 2 — Garimpeiros
02
Validação de E-mail
Verificação via ZeroBounce API antes de qualquer ação. E-mail inválido → descarte imediato. Nunca enviar para endereços não validados.
Time 3 — Analistas
03
Aderência ao ICP
Verificação de setor, porte, cargo e região contra o perfil de cliente ideal configurado. Fora do ICP → arquivado automaticamente, não descartado.
Time 3 — Analistas
04
Score 0–100
Modelo de IA calcula score com base em 12+ variáveis: porte, setor, cargo, gatilhos de compra, presença digital, localização, dados de enriquecimento. Abaixo do mínimo → descarte.
Time 3 — Analistas
05
Redação Personalizada
Pesquisa de contexto automática (notícias recentes, LinkedIn, site institucional). E-mail gerado via Claude API com menção específica à empresa. Revisão automática: sem links, sem spam words, tom de conversa.
Time 4 — Redatores
06
Cadência de 4 Toques
Dispatcher seleciona domínio saudável, respeita horário comercial do fuso do lead, aplica cadência: Dia 1 → Dia 4 → Dia 9 → Dia 16. Sem envios em fins de semana. Pausa automática se bounce detectado.
Time 5 — Cadenciadores
07
Handoff SQL → FBR-Click
Resposta positiva detectada → deal criado no FBR-Click com contexto completo, histórico de interações, score e dados da empresa. Vendedor notificado com @menção. Ciclo encerrado para o sistema.
Time 5 — Cadenciadores

05 — Guardiões & Mail Server

O protocolo que protege os domínios

A reputação dos domínios é a fundação de toda a operação. Sem domínios saudáveis, não há prospecção. O Time 1 — Guardiões — opera 24/7 para garantir que cada domínio mantenha score de reputação elevado e bounce rate abaixo de 2%.

Regra de ouro: domínio novo nunca sai do forno direto pro fogo Todo novo domínio passa por 30 dias de aquecimento interno — somente trocas entre contas do próprio sistema — antes de tocar em qualquer lead externo. Queimar um domínio no primeiro dia é a falha mais cara e mais evitável do processo.
Fase Período Volume / dia Atividade
Fase 1Dias 1–30Interno apenasTroca de e-mails entre contas do próprio sistema FBR
Fase 2Dias 31–6010–20 e-mailsPrimeiros contatos externos — somente leads com score mais alto
Fase 3Dias 61–9030–50 e-mailsVolume controlado com cadências completas de 4 toques
Fase 4Dia 90+50–100 e-mailsOperação plena com monitoramento contínuo de reputação
Regras do Dispatcher — Time 5

Sempre fazer

  • Selecionar domínio com sends_today < daily_limit
  • Respeitar fuso horário do lead
  • Registrar todo envio no audit log
  • Notificar Guardião se bounce detectado

Nunca fazer

  • Enviar em fins de semana
  • Usar domínio em fase 1
  • Enviar se bounce > 2%
  • Ignorar blacklist detectada

E-mail proibido

  • Links no primeiro toque
  • Anexos de qualquer tipo
  • GRÁTIS / PROMOÇÃO / CLIQUE AQUI
  • Mais de uma pergunta por e-mail

E-mail obrigatório

  • Contexto específico da empresa
  • 3 a 5 parágrafos, não mais
  • Um único CTA claro
  • Tom de conversa, não de vendas
06 — Agentes OpenClaw FBR

Como os agentes do FBR-Leads funcionam

Todo agente do FBR-Leads segue o padrão OpenClaw obrigatório da Bíblia FBR: definido por exatamente 7 arquivos Markdown versionados em Git, sem exceção. Cada agente trabalha em uma única tarefa por vez — foco total, sem paralelismo interno.

Princípio: n8n orquestra, OpenClaw executa O n8n (instância dedicada FBR-Leads) decide quando e por que um agente é chamado. O OpenClaw Gateway decide o que o agente faz quando chamado. Os dois não se substituem — são camadas complementares e obrigatórias.
Os 7 Markdowns obrigatórios — exemplo: Redator Principal
# agents/redator-principal/ — 7 arquivos obrigatórios SOUL.md # Carregado PRIMEIRO. Nada sobrescreve. → Identidade: "Sou o agente de redação de prospecção da FBR." → Restrições: nunca gerar conteúdo enganoso, nunca prometer o que a empresa não pode entregar, sempre revelar contexto real. → Ética: e-mails que se passam por humanos precisam de aprovação. IDENTITY.md # Perfil visível no FBR-Click → Nome: FBR-Leads-Redator → Role: Outbound Email Writer Agent → LLM primário: Claude API (OBRIGATÓRIO — Ollama não é suficiente) → LLM fallback: GPT-4o (somente se Claude indisponível) TASKS.md # Uma tarefa por vez → receber_lead_qualificado → acionar_pesquisador_contexto → gerar_email_personalizado (Claude API obrigatório) → revisar_com_agente_revisor → gerar_variante_a_b → entregar_para_dispatcher AGENTS.md # Limites de autonomia → Aprovação humana OBRIGATÓRIA para: - Alterar tom padrão de uma campanha inteira - Usar nome de concorrente do lead no e-mail - Enviar para C-Level de empresa > 500 colaboradores → Canal: #leads-qualificados no FBR-Click MEMORY.md # Atualizado ao final de sessões relevantes → Padrões de e-mail que geraram resposta positiva por setor → Palavras e frases com alta taxa de resposta por ICP → Contextos de empresa que funcionaram (gatilhos) TOOLS.md # APIs e ferramentas disponíveis → FastAPI interna (leads, campanhas, ICPs) → Claude API (geração de e-mail — obrigatório) → Pesquisador de Contexto (agente via n8n) → Revisor (agente via n8n) USER.md # Contexto da empresa → Facebrasil vende espaços publicitários e serviços digitais → Foco: empresas brasileiras nos EUA → Tom: consultivo, direto, sem jargão de marketing
07 — Integração FBR-Click

Como SQLs chegam ao time de vendas

O Cadenciador Bot é o agente OpenClaw registrado como membro do FBR-Click. Ele opera no canal #leads-qualificados e é responsável por todo o handoff — criação do deal, postagem de contexto e notificação do vendedor. O vendedor nunca precisa abrir o dashboard do FBR-Leads.

Fluxo completo: resposta do lead → deal no FBR-Click em < 5 minutos Monitor de Respostas detecta resposta positiva → Cadenciador Bot aciona handoff → Deal criado com contexto completo → Canal do deal aberto → @menção ao vendedor responsável → Ciclo encerrado para o sistema.

Feedback loop FBR-Click → FBR-Leads
✅

deal.won

Marcar lead como convertido. Registrar em intelligence_reports. Reforça padrões do ICP e scoring — Time 6 aprende o que funcionou.

💰

deal.lost (preço)

Marcar lost com razão. Ajusta peso de variáveis de budget no modelo de scoring. Sistema passa a priorizar empresas com maior capacidade.

🎯

deal.lost (não era decisor)

Refina filtragem de cargos no ICP. Ajusta o Analista de ICP para rejeitar leads sem poder de decisão no setor identificado.

🔕

deal.lost (sem resposta)

Otimiza padrões de mensagem para o Redator. Time 6 identifica qual toque ou abordagem não ressoou e sugere variantes de teste A/B.

08 — Dashboard Interno

Funcionalidades do painel de controle

O dashboard é uma ferramenta interna para o gestor — não para o vendedor. Monitora saúde de domínios, pipeline de leads, status de agentes e resultados de campanhas em tempo real via WebSocket e SSE.

📊

Overview — /dashboard

Métricas principais, saúde dos domínios, funil de leads em tempo real e últimos SQLs gerados. Atualizado a cada 60 segundos automaticamente.

📧

Saúde dos Domínios — /domains

Score, fase de aquecimento (1–4), sends_today/daily_limit, bounce_rate e status de blacklist por domínio. Alertas automáticos se bounce > 2%.

🔭

Pipeline de Leads — /leads

Funil visual por estágio, tabela com ordenação e filtros, exportação CSV e painel lateral com dados completos do lead e histórico de interações.

🎯

Configuração de ICP — /icp

Formulário no-code: setores, portes, cargos, regiões, palavras-chave e score mínimo. Após salvar, Garimpeiros iniciam coleta em até 30 minutos.

🤖

Status dos Agentes — /agents

Feed de logs em tempo real via SSE. Kill switch por agente com confirmação. Badge online/offline/erro. Histórico de ações dos últimas 24h.

📬

Campanhas — /campaigns

Wizard de criação em 3 etapas: ICP → Redator → Cadenciador. Monitoramento de taxa de abertura, clique, resposta e bounce por campanha ativa.

📈

Relatórios — /reports

Relatório semanal automático gerado pelo Time 6. Padrões identificados, sugestões de ajuste de ICP e mensagens. Exportação CSV disponível.

⚡

Tempo Real via WebSocket

Páginas de domínios e agentes usam WebSocket persistente — atualizações sem refresh. Leads e campanhas usam SSE para streaming de logs.
