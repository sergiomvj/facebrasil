O que é o FBR·Sales?

O FBR-Sales é o Departamento Comercial Digital do Grupo Facebrasil — o sistema que assume o cliente quando ele sai do FBR-Click e garante que cada centavo de publicidade entre, seja rastreado e auditado. Versão 1.0 cobre todos os ativos de mídia do grupo.

FBR-Sales no mapa do grupo

FBR-Click conduz o prospect. FBR-Sales assume quando ele vira cliente e opera a receita. Cada sistema tem escopo claro — o contrato entre eles é REST API + JWT, sem acesso cruzado a banco de dados.

Sistema Direção Relação com FBR-Sales Protocolo
FBR-Click
 → entrega clientes CRM do grupo. Conduz o prospect até fechar. Ao virar cliente, transfere ao FBR-Sales via evento de handoff. REST API · JWT
FBR-Finance
 ↔ pagamentos FBR-Sales consome a API do Finance para reconciliar pagamentos recebidos com os valores esperados nos contratos. REST API · JWT
FBR-Design
 ↔ criativos Todo criativo publicitário passa pelo fluxo FBR-Design antes de ir ao ar. Agente Auris coordena a aprovação via API. REST API · JWT
FBR-Assets
 ↔ ativos de mídia Registra e gerencia os ativos de mídia do grupo (blogs, revistas, YouTube). O FBR-Sales consulta inventário e metadados de cada publicação para compor o rate card e o media kit. REST API · JWT
Produtos do Grupo
 ← consome API Blogs, revistas e canais YouTube consomem a Ad Network API com API Key própria para comercializar espaços. API Key pública
Players Externos
 ← acordos e receita AdSense, Taboola, Ezoic, Mediavine, Mgid, Outbrain, YouTube Ads. Pipeline de parcerias e controle de receita. API por parceiro

Como os sistemas se falam

FBR-Sales nunca acessa banco de dados de outro sistema. Toda comunicação é REST API + JWT. O FBR-Click é o canal de notificação humana — todas as aprovações e alertas chegam lá.

// fluxo principal: prospect → receita auditada
FBR-Click → FBR-Sales → FBR-Finance · REST + JWT · Assíncrono
STEP 01 · FBR-CLICK
Handoff do Cliente
Prospect convertido em cliente pelo CRM. Evento de handoff enviado ao FBR-Sales com histórico de negociação.
→
STEP 02 · FBR-SALES
Pipeline Ativo
Maia assume o cliente, agenda onboarding, cria lançamento de receita pendente e ativa o monitoramento do Vigil.
→
STEP 03 · APROVAÇÃO
Proposta Aprovada
Nexus gera draft → notificação no FBR-Click → Sergio ou Marco aprovam → proposta enviada ao parceiro.
→
STEP 04 · FBR-SALES
Parceria Ativa
Contrato assinado, onboarding técnico concluído, parceiro em estado ACTIVE. Agente Ledger aguarda primeiro pagamento.
→
STEP 05 · FBR-FINANCE
Receita Auditada
Pagamento recebido no Finance → webhook para FBR-Sales → Ledger reconcilia → status muda para PAID → audit log.

API FBR-Sales

API RESTful versionada em /api/v1/ com autenticação via proxy Next.js (iron-session → X-User-Id). Endpoints de Ad Network autenticados por API Key por publicação. Audit trail em todas as operações críticas.

MétodoEndpointDescrição
GET/partnersLista parceiros com filtro por status, tipo e período. Paginação cursor.
POST/partnersCria novo parceiro no pipeline com status PROSPECT.
GET/partners/:idDetalhe do parceiro com histórico de pipeline_events, receitas e documentos.
POST/partners/:id/advanceAvança estágio no pipeline. Requer campo approved_by para transições críticas.
POST/partners/:id/contractUpload do contrato assinado (multipart). Registra URL e aprovador humano.
GET/ad-spacesLista espaços disponíveis com ocupação e CPM dinâmico. Autenticado por API Key.
POST/campaignsCria campanha com status pending_approval. Aguarda aprovação humana para ativar.
PUT/campaigns/:id/approveAprovação humana da campanha. Dispara envio de criativo ao FBR-Design.
GET/publications/:id/rate-cardRate card dinâmico da publicação. CPM ajustado pela ocupação atual do inventário.
GET/publications/:id/media-kitGera media kit via agente Quill. Retorna URL do PDF gerado pelo FBR-Design.
GET/revenueDashboard de receita consolidada com filtros por parceiro, produto e período.
POST/revenue/reconcileDispara reconciliação com FBR-Finance. Compara expected vs received e atualiza status.
GET/revenue/forecastProjeção de receita para 30/60/90 dias via agente Pulse (média móvel + contratos).
GET/revenue/calendarCalendário de vencimentos por parceiro com status (pendente, pago, atrasado).
GET/audit-logLog de auditoria completo. Append-only — sem DELETE nem UPDATE nesta tabela.
