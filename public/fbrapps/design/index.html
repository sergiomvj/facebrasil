00 — Visão do Projeto

Por que este projeto existe

"Uma empresa com dezenas de sites, blogs e clientes ativos não pode depender de um designer humano para cada cartão de visita, post de Instagram ou banner de campanha. O design gráfico padronizado deve ser automatizado — com consistência de marca, qualidade visual e velocidade de entrega."

O FBR-Design é o sistema de automação gráfica da FBR. Recebe um briefing estruturado, aciona dois agentes OpenClaw especializados em sequência — o Compositor transforma o briefing em um material gráfico renderizado; o AssetFinder garante que cada peça tenha imagens de qualidade e licença adequada. O output são arquivos PNG, JPG e PDF prontos para uso.

O sistema segue integralmente a Bíblia FBR v1.0: agentes OpenClaw com 7 Markdowns versionados em Git, orquestração via n8n dedicado, cascata de LLMs com Ollama → Claude → GPT-4o, e toda interação humana centralizada no FBR-Click. Cada agente opera em uma única tarefa por vez — foco total, sem paralelismo dentro do mesmo agente.

02 — Fluxo de Produção

Como uma peça gráfica é processada

Todo trabalho segue um pipeline determinístico com aprovação humana obrigatória. O Compositor seleciona o template, compõe o HTML e renderiza; o AssetFinder garante que as imagens certas cheguem com licença adequada. O Supervisor aprova antes da entrega.

01

Recepção do Briefing

O operador preenche o wizard em 4 etapas: tipo de material e cliente, conteúdo textual e tom, preferências visuais e referências, e upload de assets do cliente (logo, fotos, documentos). O brand kit do cliente é carregado automaticamente.

02

Agente Compositor — Interpretação e Seleção de Template

O agente FBR-Design-Compositor é acionado pelo n8n. Ele interpreta o briefing, realiza busca semântica no banco de templates via PostgreSQL (tsvector), seleciona o template de maior score de compatibilidade e mapeia os campos do briefing para as variáveis do template. LLM primário: Ollama. Fallback: Claude API → GPT-4o.

03

Agente AssetFinder — Busca e Geração de Imagens

O agente FBR-Design-AssetFinder é acionado via n8n pelo Compositor. Ele busca assets visuais em cascata: Unsplash → Pexels → Pixabay (score ≥ 0.8 em cada). Se nenhum banco gratuito atender, gera via MidJourney — neste caso, flag de aprovação obrigatória é ativada.

04

Composição HTML + Brand Kit

O Compositor carrega o HTML do template, injeta todas as variáveis — textos, imagens, cores do brand kit, fontes, logo do cliente — e valida o HTML resultante. O brand kit do cliente é inviolável: cores, tipografia e identidade visual aplicados automaticamente em 100% das peças.

05

Renderização — HTMLCSStoImage + Bannerbear

O HTML composto é enviado para a HTMLCSStoImage API, que retorna PNG/JPG/PDF de alta qualidade. Se falhar, Bannerbear atua como fallback de renderização. As variantes automáticas (Feed, Story, LinkedIn, Banner, Thumbnail) são geradas a partir da mesma composição.

06

Armazenamento e Notificação

Os outputs são salvos no MinIO (3 buckets: outputs, uploads, templates). O job é atualizado no PostgreSQL com status aguardando_aprovacao. Um alerta é criado na tabela alerts com fbr_click_notified = false.

07

Aprovação & Entrega via FBR-Click

O agente posta no canal #fbr-design do FBR-Click como membro ativo — nunca como webhook anônimo. O Supervisor visualiza o preview, aprova, solicita ajuste ou rejeita com nota. Após aprovação, o operador faz o download de todas as variantes. fbr_click_notified marcado como true.

05 — Agentes OpenClaw FBR

Como os agentes do FBR-Design funcionam

Todo agente do FBR-Design segue o padrão OpenClaw obrigatório da Bíblia FBR: definido por exatamente 7 arquivos Markdown versionados em Git, sem exceção. Cada agente trabalha em uma única tarefa por vez — foco total, sem paralelismo interno. O sistema integra com os agentes — nunca os implementa internamente.

Princípio: n8n orquestra, OpenClaw executa O n8n (instância dedicada FBR-Design) decide quando e por que um agente é chamado. O OpenClaw Gateway decide o que o agente faz quando chamado. Os dois não se substituem — são camadas complementares e obrigatórias. O backend nunca chama LLM diretamente — toda inferência passa pelo OpenClaw Gateway.
Agentes do MVP
🎨

FBR-Design-Compositor

Agente de pipeline principal. Recebe o briefing, faz busca semântica de templates, mapeia variáveis, aciona o AssetFinder via n8n, compõe o HTML com o brand kit do cliente, aciona a renderização e notifica o FBR-Click. Opera do briefing ao output renderizado em uma única execução.

🔍

FBR-Design-AssetFinder

Agente especializado em busca e geração de assets visuais. Cascata obrigatória: Unsplash → Pexels → Pixabay (gratuitos, sem copyright). Se nenhum banco retornar score ≥ 0.8, gera via MidJourney — com flag de aprovação obrigatória. Registra origem, licença e crédito de cada asset na tabela assets.

Os 7 Markdowns obrigatórios por agente

SOUL.md

  • Carregado primeiro
  • Nada sobrescreve
  • Valores e ética
  • Restrições absolutas

  • Nome e role

  • Fatos duráveis
  • Padrões aprendidos

TOOLS.md + USER.md

  • Ordem de prioridade

Processo de criação de agentes (6 passos obrigatórios)
1

Owner define propósito e limites

Responsabilidades claras, lista de ações autônomas e lista de ações que requerem aprovação humana. A lista de aprovações nunca pode estar vazia.

2

Criação do repositório Git com os 7 Markdowns

SOUL.md é preenchido primeiro. Nenhum agente é ativado sem todos os 7 arquivos presentes e completos.

3

Pull Request obrigatório

O Arquiteto FBR revisa os 7 Markdowns antes de qualquer ativação. Nenhuma exceção, nem para protótipos.

4

Registro no FBR-Click

Admin registra a URL do repositório Git na tabela agents com openclaw_agent_id e git_repo_url. Não há endpoint de criação no painel.

5

Período de observação — 7 dias

Monitoramento ativo com kill switch pronto. Qualquer comportamento anômalo resulta em desativação imediata.

6

Aprovação formal de operação plena

Owner assina a aprovação formal. Apenas após este passo o agente opera sem supervisão ativa.

06 — Recursos do Sistema

Funcionalidades do MVP

📋

Briefing Wizard em 4 Etapas

Formulário guiado que adapta os campos ao tipo de material selecionado. Um briefing de cartão de visita pede nome, cargo e contato; um de post social pede título, chamada e CTA. Brand kit carregado automaticamente pelo cliente.

🧠

Busca Semântica de Templates

O agente seleciona o template mais adequado via tsvector do PostgreSQL. Parâmetros de match: categoria, subcategoria, estilo visual, setor indicado. Retorna top 3 candidatos com score de compatibilidade antes de selecionar.

🎨

Brand Kit Engine

Cada cliente tem cores primária/secundária, tipografia, logo e guidelines configurados. O brand kit é aplicado automaticamente em 100% das peças. Inviolável: o agente não pode substituir o brand kit sem aprovação explícita do Supervisor.

🖼️

Assets em Cascata

1º busca em Unsplash, Pexels e Pixabay (gratuitos, sem copyright). Score mínimo de 0.8 para aceitar. Se nenhum banco atender, gera via MidJourney com flag de aprovação obrigatória. Assets do cliente têm prioridade absoluta.

⚡

Renderização HTML → Imagem

Templates em HTML/CSS puro são renderizados via HTMLCSStoImage API em PNG/JPG/PDF de alta qualidade. Bannerbear como fallback. Sem dependência de ferramentas de design proprietárias — templates versionados em Git.

📐

Variantes Automáticas

A partir de uma única composição aprovada, o sistema gera automaticamente todas as variantes: Feed 1:1, Story 9:16, LinkedIn, Banner Web e Thumbnail. Cinco formatos entregues por job, sem esforço adicional do operador.

✅

Aprovação via FBR-Click

O agente posta o preview no canal #fbr-design como membro ativo — nunca como webhook anônimo. O Supervisor aprova ou rejeita com nota. Toda aprovação dispara confirm-dialog no frontend. Alertas rastreados com fbr_click_notified.

🔒

Controle de Acesso por Role

Três roles: Operador (submete briefings, vê seus próprios jobs), Supervisor (aprova, vê todos os jobs), Admin (gerencia templates, brand kits e clientes). Isolamento de dados via dependency injection — sem acesso cruzado.

🎛️

Painel de Templates (Admin)

Galeria com busca semântica em tempo real (debounce 500ms). Templates criados em HTML/CSS puro, versionados em Git. Futuro: ingest visual via Claude Vision para analisar referências do Behance/Dribbble e gerar templates equivalentes.

📊

Dashboard & Analytics

Métricas de produção: jobs por período, taxa de aprovação, tempo médio de processamento, templates mais usados, distribuição de uso por camada LLM (ollama / claude / gpt4o). Gráficos via Recharts.
