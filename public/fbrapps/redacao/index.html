00 — Visão do Projeto

Por que este projeto existe

"Um imigrante brasileiro em Orlando não pode esperar que um repórter humano cubra a câmara municipal local, o acidente na I-4 ou a mudança no salário mínimo de Florida. Agentes virtuais assumem esse papel — 24 horas por dia, 50 cidades, zero jornalistas contratados."

O FBR-Redacao é a redação virtual do Facebrasil. Monitora fontes locais em dezenas de cidades americanas, produz artigos contextualmente relevantes para a comunidade brasileira, adiciona mídia, revisa editorialmente, publica e distribui — tudo de forma autônoma, com dois operadores humanos supervisionando via painel interno.

O sistema segue integralmente a Bíblia FBR v1.0: seis agentes OpenClaw com papéis editoriais distintos, orquestração via n8n dedicado, cascata de LLMs com Ollama → Claude → GPT-4o, e toda interação humana centralizada no FBR-Click. Cada agente opera em uma única tarefa editorial por vez — foco total, cadeia de responsabilidade clara.

02 — O Problema

Por que correspondentes humanos não funcionam nessa escala

A diáspora brasileira nos EUA se distribui por mais de 50 áreas metropolitanas diferentes. Cada comunidade tem suas particularidades — o imigrante em Orlando é diferente do de Boston, e as notícias relevantes para cada grupo são igualmente distintas.

01

Custo inviável de correspondentes humanos

Cobrir 50 cidades com repórteres humanos custaria dezenas de milhares de dólares por mês — inviável para qualquer publicação focada em nicho imigrante. O Facebrasil tem 16 anos e cobre poucas cidades com qualidade, sem escala.

02

Notícias genéricas não servem ao imigrante

Traduzir notícias americanas para o português não é suficiente. O imigrante precisa entender o impacto prático no seu dia a dia: "o que isso muda pra mim, que moro aqui com visto de trabalho e mando dinheiro pro Brasil todo mês?"

03

Velocidade: notícias locais exigem resposta em horas

Uma mudança na lei de aluguel de Florida, uma operação da ICE em Newark, uma greve no metrô de Boston — cada uma dessas notícias precisa de cobertura em horas, não em dias. Sem automação, isso é impossível com equipe pequena.

04

Consistência editorial em múltiplos canais

O mesmo artigo precisa virar post web, vídeo para YouTube, resumo para WhatsApp e Reel para Instagram — com formatos, comprimentos e tons diferentes. Fazer isso manualmente para 50 cidades é humanamente impossível.

03 — Pipeline Editorial

A cadeia de seis agentes

O FBR-Redacao é construído em torno de uma cadeia editorial linear com revisão automática. Cada agente tem um papel único e entrega para o próximo — exatamente como uma redação humana, mas executada a 100x a velocidade e a 1/100 do custo.

Princípio: n8n orquestra, OpenClaw executa O n8n (instância dedicada FBR-Redacao) decide quando e por que cada agente é chamado — ciclos de coleta, timeouts, retry de falhas. O OpenClaw Gateway executa o que o agente faz quando chamado. Os dois não se substituem — são camadas obrigatórias e complementares.
AGENTE 01
Coletor
Varre RSS feeds, NewsAPI, GNews e redes sociais por ciclo configurado
Desduplicação por hash de similaridade via Redis
Pontuação de relevância por região e impacto no imigrante
Enfileira apenas conteúdo novo para o Agente Jornalista
↓
AGENTE 02
Jornalista
Pesquisa contexto e background da headline
Traduz e contextualiza culturalmente para o imigrante brasileiro
Escreve artigo em Markdown: título, subtítulo, corpo, fontes
Seção obrigatória: "O que isso muda pra você"
Valida estrutura via Pydantic — nunca texto livre como dado estruturado
↓
AGENTE 03
Arte
Busca imagens licenciadas: Unsplash → Pexels (cascata)
Gera imagem via Stable Diffusion local se nenhuma encontrada
Cria thumbnail com template visual da cidade/região
Armazena no Supabase Storage e vincula ao artigo
↓
AGENTE 04
Editor Regional
Analisa qualidade editorial: tom, precisão, clareza, adequação cultural
Corrige automaticamente problemas identificados
Registra versão anterior e log de todas as alterações
Publica e sinaliza para revisão humana quando necessário
↓
AGENTE 05
Chefe de Redação
Monitora artigos publicados em busca de problemas pós-publicação
Classifica problemas: factual, cultural, legal, qualidade, imagem
Problemas graves → age automaticamente (retira do ar)
Problemas leves → gera alerta para o operador humano
↓
AGENTE 06
Moderador
Recebe contribuições do Eu Repórter: vídeo, áudio, texto, imagem
Analisa conteúdo, avalia relevância e classifica por região
Score ≥ 0.8 → aprovação automática com crédito $0,50–$2,00
Score < 0.8 → rejeição com motivo claro e específico
Contribuições aprovadas entram na fila do Coletor como pauta prioritária
04 — Agentes OpenClaw

Como os agentes do FBR-Redacao funcionam

Todo agente do FBR-Redacao segue o padrão OpenClaw obrigatório da Bíblia FBR: definido por exatamente 7 arquivos Markdown versionados em Git, sem exceção. Cada agente trabalha em uma única tarefa editorial por vez — foco total, sem paralelismo interno.

Regra absoluta: agentes vivem no OpenClaw Gateway Nenhum agente pode ser implementado internamente na aplicação (sem LangGraph no backend, sem lógica de agente no n8n). Todos os seis agentes editoriais residem exclusivamente no OpenClaw Gateway e são consumidos via API. Violações desta regra são tratadas como erros arquiteturais que exigem refatoração imediata.
Os 7 Markdowns obrigatórios — exemplo: Agente Jornalista
# agents/journalist/ — 7 arquivos obrigatórios SOUL.md # Carregado PRIMEIRO. Nada sobrescreve. → Identidade: "Sou o agente jornalista da Redação Facebrasil." → Restrições: nunca inventar fatos, nunca publicar sem seção de impacto, nunca usar linguagem tendenciosa ou sensacionalista. → Ética: artigos com implicações legais requerem sinalização obrigatória. IDENTITY.md # Perfil visível no FBR-Click → Nome: FBR-Redacao-Jornalista → Role: Editorial Journalist Agent → LLM primário: Ollama (local, via Tailscale) → LLM fallback 1: Claude API (claude-sonnet-4-6) → LLM fallback 2: GPT-4o (contingência) TASKS.md # Uma tarefa por vez → receber_headline → pesquisar_contexto → traduzir_e_culturalizar (EN→PT + contexto imigrante) → escrever_artigo (Markdown limpo, sem asteriscos desnecessários) → escrever_secao_impacto ("O que isso muda pra você") → validar_estrutura (Pydantic — nunca texto livre) → entregar_para_agente_arte AGENTS.md # Limites de autonomia → Aprovação humana OBRIGATÓRIA para: - Artigos sobre operações da ICE / deportações - Artigos com implicações legais diretas - Artigos que contrariem posição editorial anterior → Canal: #redacao-operacoes no FBR-Click MEMORY.md # Atualizado ao final de sessões relevantes → Coberturas anteriores por cidade → Tom editorial por categoria (imigração, saúde, trabalho) → Padrões de impacto mais engajantes por região TOOLS.md # APIs e ferramentas disponíveis → NewsAPI (contexto e background) → Wikipedia API (definições e histórico) → FBR-Redacao API (criar artigo no banco) → n8n webhook (sinalizar conclusão de tarefa) USER.md # Contexto da empresa → Facebrasil: 16 anos, diáspora brasileira nos EUA → Leitor típico: imigrante 25–45 anos, reside há 3–10 anos → Tom: informativo, acessível, culturalmente próximo → Prioridade: impacto prático no cotidiano do imigrante
Cascata de LLM — padrão FBR
Camada 1 — Ollama (Mac Mini M4 32GB via Tailscale) → Uso: todos os agentes em operação normal → Timeout: 15s · Fallback automático se offline > 30s → Health check: n8n verifica a cada 60s → publica no Redis Camada 2 — Claude API (claude-sonnet-4-6) # fallback → Uso: ativado automaticamente quando Ollama indisponível → Timeout: 30s · Log de fallback em agent_logs → Notificação: alerta no #redacao-operacoes via FBR-Click Camada 3 — GPT-4o API # contingência → Uso: reserva total · nunca usado como primário → Timeout: 30s · Fallback: alerta crítico no FBR-Click → Loga como alert_type = 'llm_fallback' no banco # llm.py lê status do Redis antes de cada chamada — sem latência de health check # Decisão de fallback é feita em <5ms localmente
05 — Eu Repórter

A comunidade como fonte jornalística

O Eu Repórter transforma o próprio imigrante brasileiro em correspondente. Um acidente de trânsito visto por um morador local, uma nova lei estadual que afeta trabalhadores, uma promoção relâmpago em restaurante brasileiro — o leitor envia, o agente processa, o jornalista publica.

📹

Formatos aceitos

Vídeo MP4/MOV até 500MB, áudio MP3/WAV até 50MB, texto até 5.000 caracteres e imagem JPG/PNG/WEBP até 10MB. Validação de magic bytes para segurança.

🤖

Moderação automática

O Agente Moderador analisa relevância, verifica adequação editorial e classifica por região. Score ≥ 0.8 aprova automaticamente; abaixo disso, rejeita com motivo específico e não genérico.

💵

Créditos por contribuição

Contribuições aprovadas rendem $0,50 a $2,00 em crédito para desconto na assinatura Pro. O operador humano define o valor no painel interno via slider.

🔄

Pauta prioritária

UGC aprovado entra na fila do Agente Coletor com prioridade máxima, com crédito ao contribuidor no artigo publicado. Fecha o ciclo: comunidade → redação → comunidade.

06 — Painel Interno

O que os 2 operadores enxergam

Dois operadores humanos supervisionam toda a operação via painel interno Next.js. Eles não produzem conteúdo — supervisionam, corrigem exceções e tomam decisões que os agentes não têm autonomia para tomar.

🎬

Mural Em Produção

Cards de artigos em pipeline atualizados via WebSocket em tempo real. Indicador de presença: "João está revisando este artigo." Filtros por região, status e agente.

✅

Mural Publicados

Artigos recém-publicados com ações diretas: manter, corrigir inline (editor Markdown), retornar à redação, retirar do ar ou excluir. Histórico de versões acessível.

🤖

Dashboard de Agentes

Status em tempo real de todos os seis agentes: online/offline/erro/pausado. Métricas de 24h, último heartbeat, botões de start/stop e configuração JSONB editável.

📹

Fila Eu Repórter

Player inline de vídeo e áudio, score do Agente Moderador visível, slider para definir crédito e campo de motivo de rejeição. Tudo sem sair da tela.

📊

Analytics

KPIs em tempo real: artigos produzidos hoje, agentes online, taxa de correção, UGC aprovado. Gráficos por região e por agente. Funil do Eu Repórter.

🔔

Central de Alertas

Feed em tempo real de alertas: agente offline, alta taxa de rejeição, fila UGC cheia, fallback de LLM ativado. Badge na sidebar incrementa automaticamente via WebSocket.

Tecnologias do MVP

Stack escolhida para equilibrar autonomia arquitetural (sem depender de serviços proprietários caros), velocidade de desenvolvimento e capacidade de escalar de 8 para 50+ cidades sem reescrever o core.

Frontend Next.js 15 + React 19 Painel interno dos operadores — dark mode fixo, mobile first

08 — Segurança

Checklist de segurança do MVP

O painel da Redação tem acesso total aos 6 agentes, ao banco de dados editorial e ao conteúdo UGC da comunidade. A superfície de ataque é pequena mas crítica — qualquer vazamento afeta a operação jornalística e a confiança dos leitores.

Backend
  • iron-session com httpOnly + secure + sameSite=lax — cookie nunca acessível por JavaScript
  • SESSION_SECRET mínimo 32 caracteres — exclusivamente em variável de ambiente
  • Frontend nunca fala diretamente com o backend — proxy Next.js obrigatório
  • X-Internal-Key valida origem de todos os requests entre camadas
  • X-User-Id validado via dependency injection em todas as rotas protegidas
  • RLS habilitado em 14 de 14 tabelas do Supabase
  • Pydantic validation em todos os endpoints — structured output obrigatório
  • Rate limiting: auth 5/min · upload 10/h · agente 100/h · geral 300/min
  • Upload com validação de magic bytes — extensão não é critério de segurança
  • Logs estruturados em JSON — nunca dados sensíveis, nunca stack trace ao cliente
  • Soft delete em todas as tabelas — nada é excluído permanentemente sem auditoria
  • Timeouts em todas as chamadas externas (10s APIs, 30s LLMs)
Frontend
  • user_id nunca exposto no cliente — nem em URL, nem em localStorage, nem em response body
  • Variáveis sensíveis sem prefixo NEXT_PUBLIC_ — nunca no bundle do cliente
  • Todas as chamadas via /api/proxy — o backend é inacessível diretamente do browser
  • middleware.ts protege todas as rotas /(redacao)/* com verificação de sessão
  • Ações destrutivas sempre com confirm-dialog antes de executar
  • Inputs validados com Zod antes de enviar ao proxy
  • CORS restritivo — apenas domínios do painel interno

10 — Por Que Isso Funciona

O que faz o FBR-Redacao ser defensável

🗺️

Dado local irreproduzível

A cada mês operando, o FBR-Redacao acumula o maior banco de dados de comportamento e preferência da diáspora brasileira por cidade do mundo. Esse ativo não tem preço e leva anos para replicar.

🌐

Contexto cultural como serviço

Qualquer modelo de linguagem consegue traduzir inglês para português. Nenhum consegue explicar o impacto de uma lei estadual de Florida para um imigrante com visto H-2B sem contexto editorial especializado. Este é o diferencial.

🔄

Ciclo virtuoso com UGC

O Eu Repórter cria um flywheel: mais leitores → mais contribuições → conteúdo mais rico e hiperlocal → mais leitores. Concorrentes que tentarem replicar depois encontrarão uma comunidade já engajada e co-criadora.

⚡

Escala impossível para humanos

Cobrir 50 cidades 24/7 com profundidade local é matematicamente impossível com jornalistas humanos a qualquer custo que um concorrente possa pagar. A barreira não é capital — é a integração entre IA, dados locais e contexto cultural.
